36  /** Parent class for all implementations of active contour algorithms for seeking closed contours
37  * in an image.
38  *
39  * In uriVL there is no such thing as an "active contour object." There is a Contour class
40  * (a Contour object being made up of a list of ImagePoint objects) and then there is a
41  * Hierarchy of ActiveContourDetector classes, each of which implements a specific algorithm for
42  * actively adjusting the vertices of a Contour object to fit the contour of a region in an image.
43  * All ActiveContourDetector classes proceed by summing up diverse "force" components---implemented
44  * as ContourForce objects mapped to the current Contour---such as "edge force," "internal force," etc.,
45  * to produce the "total force" that is applied to the vertices of the current contour. The fitting of
46  * the active contour is therefore modeled as the convergence of a diffusion equation, which requires
47  * a "time step" constant, a criterion for convergence (average displacement, maximum displacement,
48  * etc.), and a numerical threshold associated to that criterion.
49  *
50  * @author Clifford Manzanillo, Department of Computer Science and Statistics, URI
51  * @author Chris Allen, 3DG, URI [major revision 2009]
52  * @author jyh, 3DG, URI [major uriVL overhaul 2013]
53  */
