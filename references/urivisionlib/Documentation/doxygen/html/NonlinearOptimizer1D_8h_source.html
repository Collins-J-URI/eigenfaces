27  /** Pure virtual parent class for all 1-dimensional optimizers Its subclasses
28  * will actually implement an optimization algorithm, i.e. golden section search.
29  *
30  * I have flip-flopped for a while on a couple of questions regarding optimizers:
31  * <ul>
32  * <li>Inheritance: Is there any point at all to using inheritance for these
33  * classes, and wouldn't it be simpler to simple propose a flat list of
34  * implementations with no common parent class? All the more so since most of
35  * the functions in these classes could very well be static.
36  * I opted for a class hierarchy because this allows me in the future to define
37  * a "processing pipeline" with a generic 1d optimizer, and then pick the
38  * particular optimizer subclass to use w/o having to redefine the pipeline.</li>
39  * <li>Should an optimizer store a reference to the 1d function of the optimization
40  * problem, since all my minimize_ functions take the 1d function asone of their
41  * parameters? The answer is yes, because setting the function allows to
42  * preallocate whatever arrays/structures may be needed by the specific algorithm.
43  * Now, if the user wants to change the 1d funtion each time they solve a
44  * minimization problem, this is their problem.</li>
45  * <li>What about a set of "maximize" functions? No time, and not convinced that
46  * it's worth it. Since the user has to define & implement the function to
47  * optimize, nothing prevents implenting -f or bigNumber - f and run through
48  * minimization to get the maximum of +f.</li>
49  * </ul>
50  *
51  * @author jean-yves herve', 3D Group,
52  * Department of Computer Science and Statistics,
53  * University of Rhode Island
54  */
